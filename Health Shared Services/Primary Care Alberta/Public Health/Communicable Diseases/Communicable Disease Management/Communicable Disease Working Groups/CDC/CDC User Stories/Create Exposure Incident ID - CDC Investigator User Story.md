> [!warning] Current-state design — pending Epic-migration review
> This story documents the **current-state** practice for generating an Exposure Incident (EI) identifier (the ProvLab **EI number** / zone-specific **OI number**) so it is traceable moving forward. **There is an active review to determine whether the new solution will simplify the creation of Outbreak Identifiers versus current practice as part of the migration to Epic (Connect Care).** Until that review concludes, treat the EI-generation mechanism below as the *as-is* baseline, not a committed future-state design. The future-state direction is converging on the OMRA-generated **Outbreak ID** serving as the lab identifier — tracked as open issue [[Communicable Disease Open Issues#Design decisions & data-model gaps|CD-OI-3]].

## Background

When an outbreak is suspected, the Communicable Disease Control (CDC) investigator often needs to send laboratory samples that are **not** tied to an individual patient order — community and environmental samples (food, animal, and location testing such as sewer sampling), as well as human samples raised before the affected individuals are known. To route those samples and bring their results back to the right outbreak, the lab requires a shared identifier on the requisition. Today that identifier is the **Exposure Incident (EI) number**, requested by completing the required information on the **EI Board** intake form (a ProvLab EI number, or a zone-specific **OI number** depending on context).

The standing **agreement with the lab** is that the lab will accept the **Outbreak ID** generated by the Outbreak Application as the identifier for lab samples — for **human** samples and for **food, animal, and location** testing (e.g., sewers) alike. That ID is then carried on the corresponding lab results so the results can be **associated to a particular outbreak**. This is the convergence point between the current EI/OI practice and the OMRA [[02 - Create Outbreak Investigation - CDC Investigator User Story|Create Outbreak]] capability (O-A-4), whose AC1 provisions a system-generated **Outbreak ID** and whose desirable AC4 contemplates that ID serving as the EI / lab-requisition number ([[OMRA Database ERD]] note 10).

This story sits at the **Confirm (C)** stage of the Outbreak Value Stream: the EI ID is generated both to **confirm** that there is an outbreak situation (raising lab work before a declaration) and to **monitor** an existing one. Connect Care (Epic) remains the **system of record for individual cases** and for patient-specific lab orders — the EI ID exists precisely for the **non-patient-specific** requests that those orders do not cover (see [[CLAUDE-OMRA]] data-ownership boundaries).

## User Story

- **As a** CDC Investigator (Communicable Disease Control)
- **I need** the ability to request the generation of an Exposure Incident ID by completing the required information on the EI Board intake form
- **so that** I can track laboratory requests that are **not** patient-specific orders — to confirm there is an outbreak situation or to monitor an existing outbreak.

## Scenarios

- **Scenario A — Confirm an outbreak (happy path).** During investigation of a suspected event, the investigator requests one or more EI numbers to send community / environmental samples (e.g., a sewer sample, a food sample) to the lab. The returned results, tagged with the EI/Outbreak ID, provide enough evidence to confirm there are enough cases to warrant managing the situation as an outbreak.
- **Scenario B — Manage / monitor a declared outbreak.** Once the outbreak is declared, lab tests associated to the EI number allow **new results** to be linked back to the appropriate outbreak as part of the Communicable Disease Episode Value Stream.
- **Scenario C — Related outbreaks with multiple EI numbers.** An event spans multiple EI numbers (e.g., several sample streams or sites). The investigator must be able to associate / relate those EI numbers — and the outbreaks they belong to — with one another (see [[05 - Outbreak Assessment - CDC Investigator User Story|Outbreak Assessment]], O-A-6).
- **Scenario D — Community sample requiring an EI number for testing.** A respiratory community sample is submitted without an EI number; the lab will **not** run the Respiratory Pathogen Panel (RPP) — a required test for respiratory outbreak management — until a valid EI number is supplied, so the investigator generates one first. *(Raised by Jessica.)*
- **Scenario E — Genomic result from PHAC.** A Whole Genome Sequencing (WGS) result arrives from PHAC / the National Microbiology Laboratory; the investigator is alerted so the genomic result can be linked to the outbreak (scope boundary with the CD Episode / Abstract to be confirmed — see Desirable AC1 and Dependencies).

## Acceptance Testing

How will the user be able to say we built what they needed?

### Minimum (MVP — Must Have)

1. **Request an EI ID from the intake form.** *Given* the EI Board intake form (or its OMRA-realised equivalent), *when* the investigator completes the required information, *then* an **Exposure Incident ID** is generated that can be placed on a laboratory requisition as the sample identifier for non-patient-specific requests. [Public Health Act & Communicable Diseases Regulation — outbreak investigation and reporting; [[Create Outbreak Investigation Screen Specifications]]]
2. **One and only one EI number per outbreak tracking.** *Given* an outbreak tracking record, *when* an EI ID is associated to it, *then* exactly **one** EI ID can be linked per outbreak (1:1), so the identifier unambiguously resolves to a single outbreak.
3. **Lab accepts the EI/Outbreak ID across sample types.** *Given* a requisition for **human, food, animal, or environmental** (e.g., sewer) samples, *when* the requisition carries the EI/Outbreak ID, *then* the lab accepts that ID as the sample identifier and returns results **tagged with it**, so the results can be associated to the originating outbreak. *(Per the lab agreement.)* [[OMRA Database ERD]] note 10
4. **EI number satisfies lab test acceptance criteria.** *Given* a community sample for a test that the lab gates on an EI number (e.g., the Respiratory Pathogen Panel / RPP required for respiratory outbreak management), *when* the sample is submitted **with** a valid EI number, *then* the lab will accept and run the test; *when* submitted **without** one, *then* it is rejected. *(Raised by Jessica.)* [Public Health Act & Communicable Diseases Regulation]
5. **Communicate with the EI Board (where it is not replaced).** *Given* that the EI Board system is **not** being replaced, *when* an EI ID is generated or used, *then* the solution **communicates with the EI Board** so the lab's EI-number acceptance check is satisfied. *(Resolution of the "replace vs. integrate" question is pending the Epic-migration review — see Dependencies; if the EI Board is replaced, additional acceptance criteria are required.)* *(Raised by Jessica.)*
6. **Confirm the EI screen and data against the EI Board.** *Given* the current EI Board screen and data fields, *when* the EI-generation surface is designed/built, *then* the required intake fields and data are **confirmed against the EI Board** (if applicable), so nothing the lab relies on is lost. [[Create Outbreak Investigation Screen Specifications]]
7. **Privacy-conforming generation and audit.** *Given* the EI-generation and association actions, *when* they are performed, *then* they are scoped to the investigator's authorized team / zone / disease group per the OMRA access model and audited via `AuditLog` (actor, timestamp, before/after). Although the EI request itself is non-patient-specific, it is the parent of lab results that may carry personal health information. [[User Maintenance Screen Specifications]]; [Health Information Act (HIA), RSA 2000, c H-5 — collection, use and disclosure, [verify section]]

### Desirable (Nice to Have)

1. **Alert on a WGS lab test from PHAC.** When a Whole Genome Sequencing lab test arrives from PHAC, the system raises an alert. *Confirm* whether this is better realised as a **link from the Communicable Disease Episode and its associated Abstract** rather than an EI-generation feature (scope boundary flagged in Dependencies).
2. **Support a zone-specific OI number as well as a ProvLab EI number.** The investigator can generate a **zone-specific OI number** in addition to a ProvLab EI number, with the right variant selected for the context. *(NDOB question — confirm whether this is a variant of this story or its own story.)*
3. **Relate multiple EI numbers across associated outbreaks.** Where related outbreaks each carry their own EI number, the records can be associated so results and evidence can be viewed across the related set (pairs with [[05 - Outbreak Assessment - CDC Investigator User Story|Outbreak Assessment]], O-A-6).
4. **Pre-populate intake from the outbreak record.** When an outbreak already exists, the EI intake form pre-populates from the `Outbreak` record so the EI/Outbreak ID is provisioned without re-keying.

## Dependencies

- **Epic-migration review of Outbreak Identifier creation.** The governing open question: whether the new (Epic / Connect Care) solution will **simplify the creation of Outbreak Identifiers** versus current EI/OI practice. This story is the current-state baseline; revisit and re-status once the review concludes.
- **CD-OI-3 — Outbreak ID as the EI number.** Whether/when the system-generated `Outbreak.outbreakID` becomes the EI / lab-requisition identifier. Now informed by the lab agreement recorded here; tracked in [[Communicable Disease Open Issues#Design decisions & data-model gaps|CD-OI-3]].
- **EI Board: replace vs. integrate.** If the EI Board is **replaced** rather than integrated (AC5), additional acceptance criteria are required. Decision pending the Epic-migration review.
- **CD Episode Value Stream scope for non-notifiable results.** Open NDOB question: whether lab results that are **not** individually notifiable/reportable but important in an outbreak context (e.g., a positive entero / rhino) fall into the Communicable Disease Episode Value Stream. Resolve when CD Episode stories are explored.
- **Zone-specific OI number (Desirable AC2).** Confirm whether generating a zone-specific OI number is a variant of this story or a separate story; create/link accordingly.
- **WGS-from-PHAC scope (Desirable AC1).** Confirm whether the WGS alert belongs here or as a link from the CD Episode / associated Abstract.
- **Patient-specific orders are tracked elsewhere.** This story is explicitly for **non-patient-specific** lab requests; patient-specific orders are tracked in Connect Care (NDOB clarification).
