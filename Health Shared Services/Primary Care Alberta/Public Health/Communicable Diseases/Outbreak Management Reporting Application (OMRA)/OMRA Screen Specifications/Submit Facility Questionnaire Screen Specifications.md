## Design Specification Context

This screen lets a **Facility Operator complete and return the outbreak questionnaire** built for their facility ظ¤ either an identifiable **line list** (one row per person) or a summary **aggregate** report (period counts split staff vs. client). It is the **fill-and-return half** of the questionnaire pair: its counterpart, [[Create Facility Questionnaire Screen Specifications|Create Facility Questionnaire]] (slide 15), is the investigator-facing build/design half that defines *which* columns and questions the facility must answer. This screen is *OMRA Submit Questionnaire* (slide 17) and consumes the instrument that screen produces. In the OMRA value stream it sits at the **Exposure Trace** stage (the line-list instrument, [[Submit Facility Outbreak Line List - Facility Operator User Story|O-ET-1]]) and the **Manage** stage (the aggregate instrument, [[Submit Aggregate Outbreak Report - Facility Operator User Story|O-M-1]]), receiving and validating the completed instrument against the launching `Outbreak`.

The **primary user is the Facility Operator** ظ¤ the front-line source of the individual case detail or summary counts Public Health needs (`role 7` in the access model). The design intent is **low-friction, multi-channel submission**: an authentication-free web form, an uploadable Excel file, or a fillable PDF, so a site can report through whatever tooling it has. **Where the operator cannot submit electronically ظ¤ a phone call to PPHST, a paper form by email or fax ظ¤ an Investigator (or a PPHST Call Analyst) completes the submission as a proxy on the operator's behalf**, and can later correct a submitted instrument with a recorded rationale. The screen must therefore serve both an external, unauthenticated facility user and an internal, authenticated proxy.

Submission is **periodic and incremental**. The operator provides an initial submission and then returns to give **daily updates** ظ¤ adding newly symptomatic or newly identified individuals and updating previously reported ones ظ¤ so the screen presents the **prior day's data** and lets the operator change it rather than re-key it. A "no updates" confirmation for a reporting period is itself a valid submission, so the absence of new cases is recorded as intentional rather than a missed report. Each submission is dated and retained as **submission history** so Public Health can track the outbreak curve over time.

> **Scope ظ¤ facility and non-facility outbreak questionnaires.** This screen covers the **outbreak** questionnaire (the line-list or aggregate instrument completed and returned for an outbreak). It serves both a **facility** outbreak and a **non-facility** outbreak ظ¤ a community, environmental, or genomically linked (WGS) event with no reporting facility ظ¤ by treating **Facility as a non-mandatory field** (see Scenario 1.6 and BR 5.8). For a non-facility outbreak there is usually no external Facility Operator, so the **Investigator** completes the instrument directly. This mirrors the build-side decision on [[Create Facility Questionnaire Screen Specifications]] (2026-06-29; CD-OI-26 resolved, CD-OI-27 in [[Communicable Disease Open Issues]]). **Individual / per-case questionnaires are entered into the Epic Communicable Disease episode**, consistent with the OMRA data-ownership boundary that **Connect Care (Epic) is the system of record for individual cases** (see [[CLAUDE-OMRA]]). OMRA owns the outbreak-level instrument and its submissions; once a reported case requires an individual CD episode, the per-case questionnaire detail is captured in Epic, not here.

> **Naming caveat.** The wireframe (slide 17) is titled *OMRA Submit Questionnaire*; earlier artifacts use *AOMS*. Prefer **OMRA** in new prose. This file is titled *Submit Facility Questionnaire* to distinguish it from the build-side [[Create Facility Questionnaire Screen Specifications|Create Facility Questionnaire]] and the contact-investigation screens.

> **ERD status ظ¤ submission entities now modelled (v2.6); questionnaire-template still a gap.** As noted on [[Create Facility Questionnaire Screen Specifications|Create Facility Questionnaire]], the [[OMRA Database ERD]] still has **no table that represents a questionnaire definition** (provisional `OutbreakQuestionnaire` / `QuestionnaireItem`); the submit form resolves the columns it presents against that instrument, and that template layer **remains an open gap**. The **submission-side gaps this screen surfaced are now resolved in [[OMRA Database ERD]] v2.6 (Section 16)**: `FacilitySubmissionReceipt` (the dated acknowledgement returned to the operator, including the `isNoUpdates` "no updates" confirmation), `SubmissionValidation` + `SubmissionValidationIssue` (the **Validate** action's result, header plus per-issue detail), and `SubmissionFileUpload` (staging for an uploaded Excel/PDF before its rows are mapped to `LineListPerson` / an aggregate snapshot). The submission itself is still persisted as a `FacilityLineList` (+ `LineListPerson`, Section 5) or `FacilityOutbreakAggregateReport` (Section 8). The field table maps each completed field to the column it populates; the receipt/validation/upload columns now reference the Section 16 tables, and the questionnaire-template layer is the remaining gap. See Dependencies.

## Wireframe

The PowerPoint version is *OMRA Submit Questionnaire* (slide 17) in [Alberta Outbreak Management Reporting Application (OMRA) Design.pptx](https://albertahealthservices.sharepoint.com/teams/M365TCCNotifiableDiseaseProjects/Shared%20Documents/General/Solution%20Design%20Documents/Outbreak%20Management%20Application/Alberta%20Outbreak%20Management%20System%20Design.pptx). It will be replaced by a screen-shot of the most recent wireframe once folded into the deck.

The screen carries the facility-side actions confirmed on the slide ظ¤ **Save**, **Download Excel**, **Download .pdf**, **Upload File**, **Submission Date**, **Submission History**, and **Validate** ظ¤ which establish that the operator can complete the instrument three ways (in the web page, by Excel, or by fillable PDF), date the submission, see prior submissions, and validate before saving. Its layout has three zones:

- **Outbreak / facility context (header, read-only)** ظ¤ Outbreak ID, Outbreak Name, Location (facility name, address, phone), Outbreak Setting and Disease, carried through from the [[Create Outbreak Investigation Screen Specifications|Create Outbreak Investigation]] record, plus the **facility contact person** and the **affected unit/area**. A **Submission Date** for the period being reported and a **Submission History** picker that reloads a prior submission.
- **Questionnaire body (centre)** ظ¤ the columns defined on [[Create Facility Questionnaire Screen Specifications|Create Facility Questionnaire]], pre-populated with the **prior day's data** for editing:
  - in **line-list** mode, the clinical grid (*Demographics / Role ┬╖ Signs / Symptoms ┬╖ Lab ┬╖ Result ┬╖ Prophylaxis ┬╖ Vaccinated*) and the outcome grid (*Hospital ┬╖ Admitted ┬╖ Cause of Death ┬╖ Date*), one row per person, with **Add Person**, **Update**, and a **No updates** affordance for the period; and
  - in **aggregate** mode, the dated count snapshot split staff vs. client (symptomatic, lab-confirmed, hospitalized, deaths) with the population-at-risk and predominant-symptom-category fields.
- **Submission actions (footer)** ظ¤ **Upload File** (ingest an Excel/PDF rather than key in the grid), **Download Excel** / **Download .pdf** (retrieve the blank or pre-filled template), **Validate** (run the quality checks before commit), and **Save** (persist the submission and return a receipt).

## Acceptance Criteria Specifications

1. **Scenario Conditions** ظ¤ What scenarios have been defined that may affect the specifications.

   1. *Three submission channels, one instrument.* The same questionnaire can be completed three ways and the screen must support each:
      - **Web page** ظ¤ the operator (or investigator proxy) sees the **prior day's data** pre-loaded and updates it with the new day's data (new/updated people in line-list mode, new dated counts in aggregate mode).
      - **Spreadsheet** ظ¤ the operator **uploads an Excel file** (the schema-aligned template supplied by [[Create Facility Questionnaire Screen Specifications|Create Facility Questionnaire]]), which is **ingested** into the line list rather than re-keyed ([[Submit Facility Outbreak Line List - Facility Operator User Story|O-ET-1]] Scenario D / Desirable AC2).
      - **Fillable PDF** ظ¤ the operator completes and returns a fillable/printable PDF, used as the paper/fax fallback an Investigator may transcribe ([[Submit Facility Outbreak Line List - Facility Operator User Story|O-ET-1]] Scenario E).
   2. *Line list vs. aggregate.* The instrument's **mode** (set on the build side) determines which body the operator sees ظ¤ identifiable per-person rows (O-ET-1) or summary counts split staff/client (O-M-1) ظ¤ and what validation applies.
   3. *Operator vs. proxy.* The primary submitter is the **Facility Operator** via an authentication-free path; an **Investigator** or **PPHST Call Analyst** may submit or correct on the operator's behalf from a phone/paper report, recording who actually keyed the submission.
   4. *Initial vs. daily vs. "no updates".* A submission may be the first for an outbreak, a daily update to a prior submission, or a "no updates" confirmation for a quiet period.
   5. *Pre-EI / pre-Outbreak.* A facility may report before CDC assigns the Outbreak/EI number; the submission is captured and associated to the `Outbreak` once the identifier exists ([[Submit Aggregate Outbreak Report - Facility Operator User Story|O-M-1]] Scenario E).
   6. *Facility vs. non-facility outbreak.* For a **non-facility** outbreak (community, environmental, or WGS-linked cluster), the instrument is completed with **no facility attached**: the facility header context is optional, location is free text, and there is typically no external operator ظ¤ the **Investigator** keys the submission directly rather than via the authentication-free operator path. Validation must not require facility identity (BR 5.8). This is the home for the non-facility / WGS-candidate line list (see [[Outbreak Assessment - CDC Investigator User Story|O-A-6]] AC6; CD-OI-23 / CD-OI-27).

2. **Functional Behaviour** ظ¤ What (if any) business logic needs to be created for this.

   1. **Load and edit the prior submission.** On open, the screen resolves the instrument's column set and **pre-loads the most recent submission's data** for the reporting period so the operator changes what has changed rather than re-stating the whole list. Line-list rows distinguish **added** from **updated** via `LineListPerson.lineListPersonDateAdded` vs. `lineListPersonDateUpdated`; aggregate updates create a **new dated snapshot** (`FacilityOutbreakAggregateReport.reportDate`) rather than overwriting the prior one.
   2. **"There will be a change" ظ¤ incremental update.** Each daily submission records the delta against the prior day: existing `LineListPerson` rows are updated, new people are added, and **newly symptomatic staff / patients** is a **calculated** increment derived from the prior submission rather than a count the operator re-keys.
   3. **Upload ingestion.** **Upload File** maps an uploaded Excel to `LineListPerson` rows against the template schema (or stages an attached PDF); a mapping/validation pass runs before the rows are committed (see Business Rules). The uploaded file needs a staging home ظ¤ an ERD gap (see context).
   4. **Validate before Save.** The **Validate** action runs the quality checks (duplicate-by-date, mode integrity, mandatory-context, registry checks) and reports issues without committing; **Save** persists the submission, writes the audit record, and returns a **submission receipt** for the period (receipt store is an ERD gap).
   5. **Registry check on identifiable rows.** Where a line-list row carries enough identity, the screen can trigger a **PHN check / ULI lookup** via `ClientRegistryTransaction` and support opening the corresponding CD episode in Epic ([[Submit Facility Outbreak Line List - Facility Operator User Story|O-ET-1]] AC14).
   6. **Proxy and correction audit.** A proxy submission records the actual submitter (`FacilityLineList.lineListSubmissionPerson` / `FacilityOutbreakAggregateReport.submittedBy`); an Investigator correction of a prior submission records a **rationale** through `AuditLog` (actor, timestamp, before/after). The rationale/comment field is an ERD gap (see field table).

3. **User Experience Considerations** ظ¤ What (if any) UX options might we have to consider as part of the build.

   1. **Authentication-free, accessible web form.** The operator path must work **without a login** and meet accessibility standards ([[Submit Aggregate Outbreak Report - Facility Operator User Story|O-M-1]] AC8) ظ¤ so identity attribution for an unauthenticated submitter must be handled by the design (e.g., a tokenised outbreak link), an open access-model question.
   2. **Show the prior day's data clearly.** Make the pre-loaded prior submission visually distinct from the current day's edits, and make **Add Person** vs. **Update existing** obvious so the operator does not duplicate a person.
   3. **Make the channel choice obvious.** **Upload File**, **Download Excel/.pdf** and the in-page grid should read as three routes to the same instrument, with the web page as the default.
   4. **Surface Validate results inline.** Validation issues (likely duplicates, missing context, mode violations) should display against the offending row/field before Save, not as a post-submit rejection.
   5. **Confirm receipt.** After Save, show a confirmation/receipt ظ¤ including for a "no updates" submission ظ¤ so the operator knows the period was reported.
   6. Keep the body visually consistent with the columns defined on [[Create Facility Questionnaire Screen Specifications|Create Facility Questionnaire]] and the other OMRA investigation screens.

4. **Data Inputs and Outputs** ظ¤ What are the data elements involved for the build object. What test data is needed to support functional testing.

   1. *Data inputs (context, read-only).* Inherited from the outbreak and the built instrument: **Outbreak ID**, **Facility** (name, address, phone, contact person), **Outbreak Setting**, **Location**, **Disease**, the **questionnaire mode** and its column set, and the **prior submission** for the reporting period.
   2. *Data captured ظ¤ line-list mode (`LineListPerson`).* Per person: identity ("platinum" fields ظ¤ `lineListPersonFirstName`, `lineListPersonLastName`, `lineListPersonDOB`, `lineListPersonGender`, `lineListPersonPHN`, `lineListPersonULI`, `lineListPersonAddress`, `lineListPersonHomePhone`/`lineListPersonMobilePhone`, **none individually mandatory**); **person type** (`personTypeID` ظ¤ client/resident vs. HCW/staff); **room/unit** (`roomID`); **signs and symptoms** (`LineListPersonSymptom` with `symptomOnsetDate`); **case classification** (`caseClassificationID`); **lab** (`isLabConfirmed`, `specimenCollectionDate`, `specimenResultDate`, `specimenResult`); **outcomes** (`isHospitalized`, `hospitalizationDate`, `hospitalName`, `dischargeDate`, `isDied`, `deathDate`, `deathRelatedToOutbreak`); and **prophylaxis** / **vaccinated** (ERD gaps ظ¤ see field table).
   3. *Data captured ظ¤ aggregate mode (`FacilityOutbreakAggregateReport`).* The dated snapshot: `reportDate`; staff vs. client `*SymptomaticCount`, `*NewCasesToday`, `*TotalCases`, `*LabConfirmedCount`, `*HospitalizedCount`, `*DeathCount`, `*RecoveredCount`; `isManualEntry = true`; plus population-at-risk, predominant symptom category and onset date (ERD gaps ظ¤ see field table).
   4. *Data outputs.* The completed submission persisted (`FacilityLineList` + `LineListPerson` rows, or a `FacilityOutbreakAggregateReport` snapshot); a **submission receipt** for the period; a **submission-history** list of prior dated submissions; and downloadable **Excel** / **PDF** renderings of the instrument.
   5. *Test data* ظ¤ (a) a **respiratory line-list daily update** (continuing-care home: pre-load yesterday's list, mark two residents hospitalized, add one newly symptomatic staff member, Save ظْ verify added-vs-updated dates and newly-symptomatic calculation); (b) an **Excel upload** of the schema-aligned template (verify ingestion and duplicate-by-date detection); (c) a **"no updates" confirmation** for a quiet period (verify receipt); (d) an **aggregate school report** (counts split staff/student, daily snapshot, web page); and (e) an **Investigator proxy correction** of a submitted line-list row with a recorded rationale (verify `AuditLog` before/after).

5. **Business Rules and Validation** ظ¤ What (if any) business logic governs this application function and what can be done to build in quality checks for data validation?

   1. **Duplicate-by-date check.** The application must be able to **confirm whether a submission is a duplicate entry for a given reporting date** ظ¤ both at the submission level (a second submission for the same `reportDate` / reporting period) and at the person level (a `LineListPerson` likely matching an already-reported individual), prompting the operator to **update rather than duplicate** ([[Submit Facility Outbreak Line List - Facility Operator User Story|O-ET-1]] Desirable AC3).
   2. **Mode integrity.** A line-list submission must carry the identifiers needed to resolve a person; an aggregate submission must carry counts only and **no identifiable person rows**. Do not let an uploaded file violate the instrument's mode.
   3. **No mandatory PII.** No single identifying field is individually mandatory, so partial records (e.g., staff without DOB or PHN) are accepted rather than blocked ([[Submit Facility Outbreak Line List - Facility Operator User Story|O-ET-1]] AC2).
   4. **Outbreak association required (or deferrable).** A submission must attach to the correct `Outbreak`; a pre-EI submission is captured and associated once the Outbreak/EI number is assigned.
   5. **Pre-populated controls and lookups.** The completed form should present pre-populated controls so the operator answers rather than re-keys: the **Facility Unit list** (`Building` / `Department` / `Room`) for the affected-area picker, a **Gender** drop-down, a **day/month** drop-down for onset/outcome dates, a **lab result +/ظêْ** control, and a **Specimen collected** checkbox (`LineListPerson.specimenCollectionDate`).
   6. **Validate gate.** The **Validate** action must run rules 1ظô3 (and registry checks where applicable) and surface issues before Save; Save is allowed once outstanding errors are resolved or explicitly acknowledged.
   7. **Versioned submission history, not overwrite.** Each dated submission is retained; a correction creates a new version / audited change rather than silently overwriting, so the outbreak curve and the data each prior submission was collected under stay interpretable.
   8. **Facility is not mandatory.** For a non-facility outbreak (Scenario 1.6), validation must not require a facility identity. Facility name/address/phone/contact and the affected unit are optional (a free-text location/area is accepted), and the facility-unit picker (BR 5.5) degrades to free text. The submission still anchors to the `Outbreak` (BR 5.4 unchanged: Outbreak association required or deferrable). See CD-OI-27 in [[Communicable Disease Open Issues]].

6. **Exception Handling** ظ¤ How will the application handle edge cases, missing data, time outs, etc.?

   1. **Missing outbreak context** (form reached without a resolvable Outbreak ID / token) ظْ capture as a **pre-EI** submission to be associated later, or block and route to support, rather than writing an unanchored submission.
   2. **Upload mapping failure** (Excel columns do not match the template, or the file violates the instrument mode) ظْ reject the file with a per-column mapping report and let the operator correct and re-upload **without losing** any data already keyed in the web form.
   3. **Duplicate-by-date detected** ظْ warn and offer **update existing** rather than creating a duplicate submission/person (BR 5.1).
   4. **Client Registry timeout / no match** on a PHN/ULI check ظْ accept the submission with the partial identity and flag the row for later validation (`ClientRegistryTransaction.transactionStatus = Timeout / NoMatch`), never blocking the report.
   5. **Output-generation failure** (Excel/PDF download or web-form save) ظْ fail the affected action without losing the entered submission; allow re-try and re-download.
   6. **Correction after submissions exist** ظْ version and record a rationale (BR 5.7) rather than overwrite.

7. **Business Semantics** ظ¤ What terminology resonates with the business for this application function.

   - "Submit Questionnaire" / "Submission"; **Line List** (identifiable, one row per person) vs. **Aggregate** (summary counts); "Submission Date" and "Submission History"; **Save**, **Validate**, **Upload File**, **Download Excel**, **Download .pdf**; "No updates" / "no change" report; "newly symptomatic" (incremental, calculated); "platinum identifiers"; **facility-acquired** vs. **community-acquired**; "Facility Operator" (the front-line submitter) and "on behalf of" (the Investigator / PPHST proxy); "EI number" (links specimen results to the outbreak).

## Technical Implementation Notes

- **Access Specifications** ظ¤ The screen has **two access paths**. The primary user is the **Facility Operator** (`role 7`), who submits through an **authentication-free** web form / Excel-upload / PDF path defined for this screen ([[User Maintenance Screen Specifications]]); identity for an unauthenticated submitter must be established by the design (e.g., a tokenised, outbreak-scoped link) ظ¤ an open access-model question. The proxy path is **Investigator** and **PPHST Call Analyst** activity ([[Document Facility Outbreak Phone Encounter - PPHST Call Analyst User Story]]), authenticated under the OMRA RBAC/ABAC model and scoped by team/zone/disease group to the outbreak being worked, and is the only path permitted to **correct** a prior submission. Confirm whether correction is restricted to Investigator-and-above.
- **Security Specifications** ظ¤ The submission **collects personal health information** ظ¤ line-list demographics including the HIA "platinum" identifiers, specimen results, and outcomes ظ¤ so this screen is the point at which PHI enters OMRA. Collection and use are governed by the **Health Information Act (HIA)** and notifiable-disease reporting under the **Public Health Act** (ss. 22, 26). All save/correct actions are audited through `AuditLog` (actor, timestamp, before/after, with the correction rationale); the authentication-free path needs particular attention to attribution, transport security and abuse prevention. PHN/ULI validation runs through `ClientRegistryTransaction` with full request/response tracking. Any disclosure or export of completed submissions should align with the disclosure-audit approach used elsewhere in OMRA (`DisclosureExportLog`).
- **Performance Expectations** ظ¤ In-page save and the duplicate-by-date / mode-integrity validation should complete within interactive targets; the Client Registry check is an external call and must degrade gracefully on timeout (ESC 6.4) rather than block the report. Excel/PDF generation and large-file ingestion may be queued. No response-time expectation beyond the application SLA; disease-reference and facility-unit lookups must be cached and index-backed.
