Design Rationale

Why Epic's ROI Web Service API?

Epic provides a proprietary REST web service specifically for creating and updating ROI requests (open.epic specification #5450, under Web Services). This is the supported API for programmatic ROI integration. The API requires a client ID obtained through Epic's app registration process.

Akana API Gateway

All calls from the IBM Integration Engine to Epic route through the Akana API Gateway, which provides authentication management, rate limiting, API governance, and audit logging at the gateway layer.

Two Response Paths

Automated (4a-6a): Pre-approved request types bypass clinician review. HSS detects the approval via polling through Akana, then pulls the cross-CIS encounter summary from Snowflake.

Clinician-prepared (4b-6b): A clinician reviews the request within Epic, prepares a response document, and HSS retrieves that document via the API once the status indicates completion.

Why Snowflake for the Automated Path?

The CI program requires an encounter summary across multiple clinical information systems, not just Epic Connect Care. Snowflake already aggregates this cross-CIS data for analytics.

Key Considerations

Client ID registration: HSS needs a registered client ID through Epic's app creation process. The full API spec (#5450) requires an open.epic login.

Patient identity resolution: PHN is optional. Epic's Patient Lookup and Identifiers Web Service (spec #5454) can resolve patient identity prior to creating the ROI request.

Denial / partial approval: Design should handle denial or requests for additional information from the clinician.

Audit and disclosure logging: Every data release must be logged with statutory authority reference, ROI Case ID, and actual data disclosed (HIA compliance).

Timeout handling: If clinician review exceeds an SLA, CMI should be notified for escalation.
