# CI RoI Spec — External Akana ↔ RIE Handoff

**Owning teams:** PHIE (Akana) — producer side · RIE (IBM ACE) — consumer side
**Sequence steps covered:** arrows 2–3 (inbound pass-through, 202), and the "hand signed payload for delivery" legs of Steps 4a / 5b / error path — per [[CI RoI Proposed Sequence Diagram]]

## Scope

This is the internal HSS handoff between the external Akana gateway and IBM ACE (the Regional Integration Engine). Akana forwards inbound requests untransformed; ACE returns the synchronous `202 Accepted` and later hands back signed webhook payloads for Akana to deliver.

## Inbound forward (Akana → ACE)

| Item | Requirement |
|---|---|
| Payload | `POST /roi-request` JSON body forwarded byte-for-byte (no transformation) |
| Headers | Akana adds gateway context (validated client cert subject, gateway request timestamp); confirm exact header names between teams |
| Request ID | **Generated by ACE**, returned in the `202 Accepted` response and relayed by Akana to CIP. Becomes the correlation/idempotency key end-to-end |
| Sync response | ACE returns `202 Accepted` immediately, before the Epic SOAP call — failures after 202 are reported via the error webhook (see [[CI RoI Sequence - All Engines Automated vs Human]], note 10) |
| ACE unavailable | Akana returns `503` to CIP without queueing — CIP retries the submission. Confirm this vs. gateway-side queueing between teams |

## Outbound handoff (ACE → Akana)

ACE formats, HMAC-signs, and disclosure-logs the payload (success, error, or clinician document), then hands it to Akana for mTLS delivery to the static CIP endpoint.

| Item | Responsibility |
|---|---|
| Payload formatting + `X-HSS-Signature` / `X-HSS-Timestamp` | RIE (ACE) |
| mTLS delivery, cert pinning, static destination config | PHIE (Akana) |
| Retry with backoff + jitter | Agree which side owns the retry loop — recommendation: ACE owns retries, Akana is a stateless delivery hop |
| Dead-letter store + operator replay | RIE (ACE) |
