A business domain is a high-level business architecture concept that allows an enterprise's business architecture to be broken down a level deeper as shown in the [[Enterprise Domain Model]].  Domain scope is defined by the value streams they are responsible for and the knowledge and value stream networks required for that domain to operate. These value streams and the associated business capability instances that support the value stream stages within that domain set the stage for the domain's enterprise architecture.

Outside the domain's scope are the value streams that form the [[Value Stream Network]] required for the domain value streams to execute both operationally and foundationally. The value stream network is important for it's own sake for assessing areas for improvement that span the organization.  Within a Domain architecture, the focus is on the value stream(s) and the business capability instances within the domains's scope and control. 

So what's the difference between an operational vs. foundational value stream? The key difference is how directly they impact the execution of the value stream vs. the foundation required for the value stream to be able to execute at all. Note that one domain's foundational value stream may be another's operational one.
1. **Foundational Value Streams** - these are usually 'back office' value streams like human resources, facility management, procurement, etc. These value streams need to be in place for the operational value streams to execute. These value streams are often intermittently triggered.
2. **Operational Value Streams** - These are the value streams that execute to generate value for a stakeholder for that particular domain. These value streams occur many times in a day, often concurrently depending on the domain's capacity.

From a scope perspective, there are aspects of the Enterprise Architecture that is unique to that domain. It's important to note that while the Enterprise Architecture documents the domain architecture and (hopefully) opportunities for improvement, it is informed by the overall busienss architecture. This can include the following:

- **[[Business Architecture]]** - beyond value streams, domains usually have unique roles and other stakeholders that may regulate that particular domain.
- **[[Application Architecture]]** - Domain-specific applications or modules of applications.
- **[[Information Architecture]]** - Domain vocabularies, regulatory reporting standards, and information exchange standards.
- **[[Technology Architecture ]]** - Usually restricted to domain-specific end user devices. Cloud or data center standards are rarely domain-specific.
- **[[Risk Architecture 1]]** - Risks and the appropriate risk mitigation plans span all aspects of the Enterprise Architecture. That said, each individual domain will need to have their own business continuity plans for their value streams.

The number of domain's that need to be defined within an organization is usually determined by the organization's size. Creating and maintaining a domain architecture requires a coordinated effort across multiple disciplines. If you're taking an Enterprise Agile approach, a multidisciplinary team that could be fed by two pizzas may be good criteria for creating a domain. This bottom up approach will look to leverage where resources are already available dedicated to a domain, yet provide a framework for collaboration across the disciplines.
